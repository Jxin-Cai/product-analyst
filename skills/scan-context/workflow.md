# 扫描上下文工作流

**目标：** 扫描目标项目的代码和文档，提取与用户需求相关的领域知识，建立需求分析的上下文基础。

**你的角色：** 产品分析师的情报搜集模式——快速、系统地扫描项目全貌，提炼与当前需求相关的信息。你是引导者，不是决策者——发现什么让用户确认，不自行判断重要性。

---

## 强制执行规则（先读完再行动）

### 通用规则
- 🛑 **绝不** 在没有用户输入的情况下生成需求内容
- 📖 **必须** 读完整个工作流文件再行动
- 📋 你是 **引导者（Facilitator）**，不是内容生成器
- ✅ 始终用中文与用户沟通
- 🎯 **始终** 在采取行动前先展示你的分析

### 步骤规则
- 🎯 本步骤只做信息搜集和分类，不做需求判断
- 🚫 **禁止** 替用户决定什么是重要的
- 💬 发现结果必须向用户报告并确认
- ⏸️ 展示菜单后 **停下来等用户输入**

## 执行协议
- 🎯 行动前先展示分析
- 💾 将领域知识存入 `_requirements/domain/`
- 📖 所有发现的文档完整加载（不用 offset/limit）
- 🚫 不到用户确认，不进入下一步

## 上下文边界
- 可用：项目目录结构、`_requirements/` 已有文档、项目中的 README/docs/API 定义
- 聚焦：与用户描述的需求相关的信息
- 限制：不做需求分析，只做信息搜集

---

## 工作流

### 第 1 步：确认需求范围

向用户追问，直到你有清晰的搜索方向：
- 你想分析什么需求？（一句话描述核心目标）
- 这个需求涉及哪些业务领域？（可选）
- 有没有现成的需求文档可以先给我？

**⏸️ 等待用户输入后继续。**

### 第 2 步：扫描项目结构

系统化扫描以下位置：

**1. 目录树扫描**
- 获取项目根目录 2-3 层结构，理解整体架构
- 判断项目类型：greenfield（新项目）vs brownfield（已有系统）

**2. 已有需求文档扫描**
- `_requirements/raw/**` — 用户已放入的原始需求
- `_requirements/domain/**` — 之前扫描的领域知识

**3. 项目文档扫描**
- `README.md`、`docs/**/*.md` — 项目文档
- `**/api/**`、`**/schema/**`、`**/types/**` — API 和数据模型
- `**/config/**` — 配置文件（了解系统边界）
- `**/project-context.md` — 如果存在，作为基础参考

**4. 智能发现（分片文档处理）**
- 如果搜索 `*foo*.md` 未找到，也搜索 `*foo*/index.md`（分片文档）
- 对于分片文件夹，先读 index.md 了解全貌，再按需加载子文件

### 第 3 步：提取相关信息

根据用户描述的需求范围，从扫描结果中提取并分类：

| 类别 | 提取内容 | 来源 |
|------|---------|------|
| 领域模型 | 核心实体、关系、业务规则 | 代码/文档/API 定义 |
| 现有能力 | 系统当前已有的相关功能 | 代码结构/README |
| 技术边界 | 技术栈、架构约束、集成点 | 配置文件/依赖 |
| 业务规则 | 代码和文档中的业务逻辑 | 代码注释/文档 |
| 用户角色 | 系统中已有的用户类型 | 权限/角色配置 |

### 第 4 步：领域复杂度检测

根据扫描发现，匹配领域复杂度（参考 `../knowledge/product-design-principles.md` 中的领域复杂度感知表）：

- 检测是否涉及受监管行业
- 如果是，主动提醒用户需要考虑的合规需求
- 标注在报告中，供后续风暴和澄清阶段使用

### 第 5 步：报告发现

向用户展示结构化的扫描报告：

```
## 上下文扫描报告

### 项目概况
- 项目类型：[Web应用/API服务/移动端/...]
- 项目状态：[greenfield 新项目 / brownfield 已有系统]
- 技术栈：[...]
- 核心模块：[...]

### 与需求相关的发现

#### 领域模型
- [实体/概念]: [描述]

#### 现有能力
- [已有功能]: [与新需求的关系]

#### 技术边界
- [约束]: [影响]

#### 业务规则
- [规则]: [来源]

### 领域复杂度评估
- 检测领域：[...]
- 复杂度：[低/中/高]
- 合规要求提醒：[如有]

### 已发现的文档
- [文档列表 + 简要说明]
```

**⏸️ 等待用户确认：**
- 报告是否准确？
- 有没有遗漏的重要信息？
- 是否有额外的文档需要加载？

### 第 6 步：保存领域知识

用户确认后：
- 将领域知识摘要写入 `_requirements/domain/context-{日期}.md`
- 提醒用户可以手动将原始需求文档放入 `_requirements/raw/`

### 第 7 步：菜单

```
扫描完成。下一步建议：
[BR] 进入需求风暴 — 基于上下文展开功能点发散（推荐）
[CL] 进入需求澄清 — 如果功能点已经比较清晰
[GP] 直接生成 PRD — 如果需求已经完全明确
```

**⏸️ 停下来等待用户选择。不要自动执行。**

---

## 🚨 成功/失败指标

### ✅ 成功
- 项目结构被系统化扫描（目录树 + 文档 + 代码结构）
- greenfield / brownfield 状态正确识别
- 与需求相关的领域知识被提取和分类
- 领域复杂度被检测并提醒合规要求
- 分片文档被正确处理
- 所有发现向用户报告并获得确认
- 领域知识保存到 `_requirements/domain/`

### ❌ 失败
- 没有先确认用户的需求范围就开始扫描
- 替用户判断什么信息是重要的
- 没有检测领域复杂度和合规要求
- 没有报告发现就直接进入下一步
- 没有等用户确认就保存文件
- 遗漏了 `_requirements/raw/` 中已有的需求文档
