# 生成 PRD 工作流

**目标：** 将澄清后的完整需求，按极简 PRD 模板写成面向双受众（人类 + LLM）消费优化的高密度文档。

**你的角色：** 产品分析师的产出模式——把发散和澄清的成果收敛成一份信息密度极高的 PRD。此刻你不再是引导者，而是精准的文档工程师。

**核心心态：** 每个字都有信息量。表格 > 段落。边界条件 > 实现细节。PRD 是所有下游工作的基础——UX 设计、架构、Epic 拆分、开发 Agent 都从这里开始。质量在这里决定一切。

## 工作流

### 第 1 步：汇总需求素材

从所有来源汇总以下信息：

| 信息类别 | 来源 | 是否必须 |
|---------|------|---------|
| 目标用户和角色列表 | 风暴/澄清产出 | ✅ 必须 |
| 核心价值和要解决的问题 | 原始需求/对话 | ✅ 必须 |
| 功能模块和功能点清单 | 风暴/澄清产出 | ✅ 必须 |
| 三条链路（正/异/逆） | 澄清产出 | ✅ P0 必须 |
| 边界条件和约束 | 澄清产出 | ✅ P0 必须 |
| 优先级标注 | 澄清产出 | ✅ 必须 |
| 领域合规要求 | 扫描/澄清产出 | 如有领域 |
| 开放问题 | 澄清产出 | 如有 |

**如果关键信息缺失**，向用户追问后再继续。不猜测，不填默认值。

### 第 2 步：生成 PRD 文档

读取 `./prd-template.md` 模板，按以下规则生成各章节：

#### 章节 1：概述

**目标用户：**
- 每个角色一行，格式 `[角色名]: [一句话描述]`
- 从风暴阶段的用户视角发散中提取

**核心价值：**
- 2-3 句话说清痛点和价值
- 不用"我们致力于"、"我们计划提供"之类的废话
- 直接说：解决什么问题 → 提供什么价值 → 与现有方案的差异

**成功标准：**
- 每条一行，必须可量化
- 分用户成功和业务成功两类
- 不接受"提高用户满意度"——要写"用户留存率 > 60% (30天)"

#### 章节 2：功能清单

按功能模块分组，每个模块一个表格。严格遵循写作规范：

```markdown
### [功能模块名]

| # | 功能点 | 正向路径 | 异常路径 | 逆向路径 | 边界条件 | 优先级 |
|---|--------|----------|----------|----------|----------|--------|
| F1 | [角色] 可以 [能力] | [1句话正常流程] | [失败场景+恢复] | [撤销机制/不可逆] | [限制/排除/极端] | P0 |
```

**写作标准：**
- 功能点列：`[角色] 可以 [能力]`，不含实现细节
- 正向路径：1 句话描述正常流程和预期结果
- 异常路径：失败场景 + 用户看到什么 + 恢复方式
- 逆向路径：可逆写撤销机制和回退状态，不可逆写 "不可逆，[原因]"
- 边界条件：数值限制、排除场景、极端情况
- 每列不超过 2 句话
- P0 功能的异常和逆向路径 **不能** 留空

**功能模块分组原则：**
- 按用户能力领域分，不按技术层分
- ✅ "用户管理"（不是"认证系统"）
- 目标 5-8 个能力领域

#### 章节 3：全局约束

**边界条件：**
- 跨功能的排除声明和限制
- 格式 `[边界]` 或 `[排除]` 前缀

**非功能约束：**
- 只写与本产品实际相关的
- 模板：`系统应 [指标] [条件] [度量方式]`
- 不堆砌通用 NFR 模板

**领域合规（如果适用）：**
- 从扫描阶段检测的领域复杂度中提取
- 只写本产品必须遵守的合规要求

#### 章节 4：开放问题

- 从澄清阶段继承的未决问题
- 生成过程中发现的新问题
- 每个问题标注影响范围（涉及哪些功能点）

### 第 3 步：风暴内容调和

**检查是否有风暴产出文档**（`brainstorm-*.md`）。如果有：

1. 加载风暴文档，提取所有功能点和想法
2. 与 PRD 交叉比对——风暴中的每个想法是否落入了 PRD 的某个章节？
3. **识别遗漏**：风暴中有但 PRD 中没有的想法，特别关注：
   - 用户体验和交互设计类想法（最容易丢失）
   - 产品调性和设计哲学类想法
   - "这个产品应该是什么感觉"类想法
4. 向用户报告："`以下风暴想法没有进入 PRD：[列表]。需要补充吗？`"
5. 如果用户要补充，加入最合适的 PRD 章节

### 第 4 步：质量自检

生成后严格自检（参考 `../knowledge/prd-writing-guide.md` 第 8 节质量自检清单）：

#### 完整性检查
- [ ] 所有用户角色都有对应的功能点？
- [ ] 每个 P0 功能点的异常和逆向路径都填了？
- [ ] 每个 P0 功能点都有边界条件？
- [ ] 领域特定的合规需求都覆盖了？

#### 信息密度检查
- [ ] 没有可以删掉而不影响理解的句子？
- [ ] 没有主观形容词（"直观"、"易用"、"友好"）？
- [ ] 没有实现细节泄露（技术栈、数据库类型、API 实现）？

#### 可追溯性检查
- [ ] 每个功能点可追溯到核心价值或用户需求？
- [ ] 功能编号连续且唯一？
- [ ] 核心价值 → 成功标准 → 功能模块 → 功能点 链条完整？

#### MECE 检查
- [ ] 功能点之间没有重叠描述？
- [ ] 没有遗漏的用户场景？

向用户报告自检结果和发现的问题。

### 第 5 步：用户确认

向用户展示生成的 PRD 全文，询问：

```
PRD 已生成。请审阅：

文档统计：
- 功能模块：[N] 个
- 功能点：[M] 个（P0: [a], P1: [b], P2: [c]）
- 开放问题：[K] 个
- 质量自检：[通过/有 X 个待处理项]

请确认：
1. 有需要修改或补充的地方吗？
2. 开放问题中哪些现在可以确认？
3. 功能清单的优先级是否准确？

[S] 保存 — 确认无误，保存 PRD
[E] 修改 — 告诉我需要调整的地方
```

**⏸️ 等待用户确认。根据反馈修改后再保存。**

### 第 6 步：保存 PRD

用户确认后保存到 `_requirements/prd/prd-{项目名}-{日期}.md`

```
PRD 已保存到：_requirements/prd/prd-{项目名}-{日期}.md

文档统计：
- 功能模块：[N] 个
- 功能点：[M] 个（P0: [a], P1: [b], P2: [c]）
- 开放问题：[K] 个

后续建议：
- 将 _requirements/prd/ 下的 PRD 文档提交到版本控制
- 开放问题需要在开发前确认
- P0 功能点可以直接用于创建架构设计和 Epic 拆分
- PRD 是所有下游工作的基础——UX、架构、开发 Agent 都从这里开始
```

## 🚨 成功/失败指标

### ✅ 成功
- 所有需求素材被汇总且关键信息无缺失
- PRD 严格遵循模板结构
- 功能清单用表格格式，六列完整
- P0 功能的异常和逆向路径无空白
- 信息密度高，无废话、无实现泄露、无主观形容词
- 可追溯性链完整（核心价值 → 成功标准 → 功能模块 → 功能点）
- 风暴内容调和完成，无遗漏的重要想法
- 质量自检通过
- 用户确认后才保存
- 领域合规要求被包含（如果适用）

### ❌ 失败
- 关键信息缺失时猜测而非追问用户
- PRD 包含实现细节（技术栈、数据库、API 实现方式）
- P0 功能的异常或逆向路径留空
- 使用主观形容词（"直观"、"易用"、"强大"）
- 功能模块按技术层而非能力领域分组
- 没有做风暴内容调和
- 没有做质量自检
- 没有等用户确认就保存
- 可追溯性链断裂
