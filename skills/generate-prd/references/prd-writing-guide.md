# PRD 写作规范

本文档定义了 PRD 的写作标准。目标：极致信息密度，面向双受众（人类 + LLM）消费优化，每个字都有信息量。

---

## 1. 双受众设计

PRD 同时服务两类读者：

### 人类读者（产品经理、利益相关者）
- 清晰的专业语言，逻辑流畅
- 从愿景到需求的自然递进
- 便于审阅和批准

### LLM 下游消费（UX 设计、架构、开发 Agent）
- `##` 级标题确保每个章节可被独立提取
- 表格格式让 LLM 能精确解析每个功能的六个维度
- 功能编号（F1, F2...）支持下游文档的精确引用
- 一致的结构和模式，便于自动化处理

---

## 2. 信息密度标准

### 消灭废话

| 废话模式 | 改写 |
|---------|------|
| The system will allow users to... | 用户可以... |
| It is important to note that... | （直接陈述事实） |
| In order to... | 为了... |
| 该功能主要用于帮助用户实现... | 用户可以... |
| 我们计划提供一个... | 提供... |
| 该系统应当能够... | 系统... |
| 在此基础上，我们还将... | 另外，... |

### 密度检验

- 每句话删掉后，如果不影响读者理解，说明这句话是废话
- 每个形容词都必须可量化，否则删除："快速" → "200ms 内"，"大量" → "1000 条"
- 不出现主观形容词："直观"、"易用"、"友好"、"灵活"、"强大"

### 反模式清单

| 反模式 | 问题 | 修正 |
|--------|------|------|
| 主观形容词 | "直观易用"不可测量 | 用具体指标："3 步内完成" |
| 实现泄露 | "用 JWT 做认证" | "支持令牌认证" |
| 模糊量词 | "多个用户"、"若干选项" | "最多 100 个并发用户"、"3-5 个选项" |
| 缺失测试标准 | "系统提供通知" | "系统在触发后 30 秒内发送邮件通知" |

## 3. 功能点表达格式

### 标准 FR 格式

```
[角色] 可以 [能力] [约束条件/边界]
```

**SMART 质量标准：**
- **Specific**：清晰、精确定义的能力
- **Measurable**：可量化，有测试标准
- **Attainable**：在约束条件内可实现
- **Relevant**：对齐业务目标
- **Traceable**：可追溯到用户需求来源

**好的例子：**
- 用户可以通过邮箱链接重置密码 [24h有效，每日限3次]
- 管理员可以批量导入用户 [CSV格式，单次上限10000行]
- 系统在订单超时30分钟后自动取消 [已支付订单除外]

**坏的例子：**
- 系统应该提供密码重置功能（缺乏边界）
- 使用 JWT + Redis 实现 token 管理（实现泄露）
- 界面应该直观易用（不可测量）

### FR 高度检验

每个 FR 应该回答"做什么（WHAT）"而不是"怎么做（HOW）"：
- ✅ "用户可以自定义外观设置" — 可以用 5 种不同技术实现
- ❌ "用户可以切换明暗主题，3 种字号，存入 LocalStorage" — 实现泄露

## 4. 表格式功能清单

PRD 的核心是功能清单表格，每行一个功能点：

```markdown
| # | 功能点 | 正向路径 | 异常路径 | 逆向路径 | 边界条件 | 优先级 |
```

### 列定义

- **#**：功能编号（F1, F2...），下游文档通过编号精确引用
- **功能点**：一句话能力声明，格式 `[角色] 可以 [能力]`
- **正向路径**：正常使用时发生什么（happy path）
- **异常路径**：出错时发生什么 + 用户如何恢复
- **逆向路径**：撤销/回退机制，不可逆标注 "不可逆，[原因]"
- **边界条件**：数值上下限、排除场景、极端情况
- **优先级**：P0（MVP 必须）/ P1（Growth 应该有）/ P2（Vision 可以有）

### 填写原则

- 每列用 1-2 句话，不展开实现细节
- 异常路径重点写"用户看到什么"和"如何恢复"
- 边界条件写数值上限/下限、明确排除的场景
- 不可逆操作在逆向路径标注 "不可逆，[原因]"
- P0 功能的异常和逆向路径不能留空

## 5. 可追溯性链

PRD 是追溯链的起点：

```
核心价值 → 成功标准 → 功能模块 → 功能点 → (未来: 验收标准)
```

**在 PRD 中必须建立：**
- 核心价值 → 成功标准的对齐
- 成功标准 → 功能模块的覆盖
- 功能模块 → 功能点的映射
- 所有功能点可追溯到用户需求

**下游影响：**
- UX 设计师读 FR → 设计交互
- 架构师读 FR + NFR → 设计系统
- 开发 Agent 读 FR → 精确实现
- QA 读 FR + 边界条件 → 生成测试用例

## 6. 结构化规范

### 标题层级

- `#` 文档标题（仅一个）
- `##` 主要章节（概述、功能清单、全局约束、开放问题）
- `###` 功能模块名 / 子章节

### 章节顺序

1. 概述（目标用户、核心价值、成功标准）
2. 功能清单（按功能模块分组的表格，非按技术层分）
3. 全局约束（跨功能的边界条件 + 非功能约束）
4. 开放问题（待确认的决策点，标注影响范围）

### 功能模块分组原则

按用户能力领域分组，不按技术层分：
- ✅ "用户管理"（不是"认证系统"）
- ✅ "内容发现"（不是"搜索引擎"）
- ✅ "团队协作"（不是"WebSocket 基础设施"）

目标 5-8 个能力领域。

## 7. 边界条件表达法

### 显式标注

用前缀标注关键限制：

```
[边界] 单个用户最多创建 100 个项目
[边界] 文件上传限 50MB，仅支持 PDF/DOCX/PNG
[边界] 本期不支持多语言，仅中文
[排除] 不处理离线场景
[排除] 不支持 IE 浏览器
```

### 全局约束 vs 功能级边界

- **全局约束**：影响所有功能的限制（性能、安全、兼容性），放在"全局约束"章节
- **功能级边界**：只影响单个功能的限制，放在功能清单表格的"边界条件"列

### 非功能约束的写法

**模板：**
```
系统应 [指标] [条件] [度量方式]
```

**好的例子：**
- 系统应在 200ms 内响应 API 请求（95th 百分位，APM 监控）
- 系统应支持 10000 并发用户（压测验证）
- 系统应保证 99.9% 可用性（工作时间，云服务商 SLA）

**只写与本产品实际相关的非功能约束**，不堆砌通用模板。

## 8. 质量自检清单

PRD 完成后，用以下清单验证：

### 完整性
- [ ] 所有用户角色都有对应的功能点？
- [ ] 每个 P0 功能点的异常和逆向路径都填了？
- [ ] 每个 P0 功能点都有边界条件？
- [ ] 领域特定的合规需求都覆盖了？

### 信息密度
- [ ] 没有可以删掉而不影响理解的句子？
- [ ] 没有主观形容词？
- [ ] 没有实现细节泄露？

### 可追溯性
- [ ] 每个功能点可追溯到核心价值或用户需求？
- [ ] 功能编号连续且唯一？

### MECE
- [ ] 功能点之间没有重叠描述？
- [ ] 没有遗漏的用户场景？
