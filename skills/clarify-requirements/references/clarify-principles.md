# 需求澄清原则

本文档定义了需求澄清阶段必须遵循的产品设计原则。用这些原则确保每个功能点的边界条件和三条链路完整。

---

## 1. 三条链路完整性

每个功能必须同时考虑三条链路：

### 正向路径（Happy Path）
用户按预期使用，一切顺利。

### 异常路径（Exception Path）
事情出错了：
- 输入非法或超出范围
- 依赖服务不可用
- 并发冲突
- 权限不足
- 超时、限流
- 部分成功（批量操作中有几个失败）

**关键问题**：失败后用户看到什么？系统状态是什么？如何恢复？

### 逆向路径（Reverse Path）
用户想撤回或回退：
- 操作可以撤销吗？
- 回退到什么状态？
- 已经产生的副作用（通知、扣款、日志）如何处理？
- 有时间窗口限制吗？
- 不可逆操作是否有二次确认？

**检验方法**：对每个功能问三个问题——"如果一切顺利？"、"如果出错了？"、"如果想反悔？"

---

## 2. 边界思维

好的需求定义不仅说"做什么"，更要说清楚：

- **不做什么**：明确排除的功能和场景
- **极端情况**：最大值、最小值、零值、空值
- **并发场景**：两个用户同时操作同一资源
- **时间边界**：操作有效期、数据保留期、重试次数上限
- **容量边界**：最多多少条记录、最大文件大小、最高并发数
- **权限边界**：谁能做什么、谁不能做什么

---

## 3. 穷举路径枚举（边缘用例猎手法）

不靠直觉找边缘场景，而是机械地穷举所有分支路径：

### 方法

对每个功能点，逐一检查：
1. **控制流分支**：每个 if/else、每个循环、每个提前返回
2. **数据边界**：零值、空值、最大值、最小值、负数、特殊字符
3. **状态转换**：每个状态之间的合法/非法转换
4. **并发竞争**：两个操作同时发生时的结果
5. **时序依赖**：操作的先后顺序影响结果的场景
6. **外部依赖**：第三方服务超时、返回异常、数据格式变化

### 输出格式

对每个发现的未处理路径，记录：
- **触发条件**：什么情况下会触发
- **潜在后果**：如果不处理会怎样
- **建议处理**：应该如何处理

---

## 4. 需求澄清追问法

对每个功能点进行结构化追问：

### 追问框架

| 维度 | 追问模板 | 目标 |
|------|---------|------|
| 边界 | "这个功能的上限/下限是什么？" | 消除模糊 |
| 异常 | "如果 X 失败了，用户看到什么？如何恢复？" | 补全异常路径 |
| 逆向 | "能撤销吗？回退到什么状态？" | 补全逆向路径 |
| 并发 | "两个人同时做这件事会怎样？" | 发现竞争条件 |
| 权限 | "谁能做？谁不能做？越权了会怎样？" | 明确权限边界 |
| 依赖 | "这个功能依赖什么外部条件？条件不满足怎么办？" | 发现外部依赖 |

### 苏格拉底式追问

不直接给答案，而是通过提问引导用户自己发现：
- "你提到了 X，这让我想到——如果 Y 情况发生呢？"
- "从用户角度看，他们期望在这种情况下看到什么？"
- "如果我是一个刁钻的测试工程师，我会怎么试图破坏这个功能？"

---

## 5. 领域复杂度感知

不同领域有不同的强制性要求，在需求分析中必须主动检测：

| 领域 | 复杂度 | 必须覆盖的需求维度 |
|------|--------|------------------|
| 医疗健康 | 高 | 数据隐私（HIPAA）、审计追踪、MFA、PHI 加密 |
| 金融科技 | 高 | 支付安全（PCI-DSS）、反洗钱（AML/KYC）、审计 |
| 政务/GovTech | 高 | 等保合规、无障碍（WCAG 2.1 AA）、数据驻留 |
| 电商 | 中 | 支付安全、库存精度、多辖区税务计算 |
| 教育 | 中 | 儿童隐私（COPPA/FERPA）、无障碍 |
| SaaS/工具 | 低-中 | 多租户隔离、数据导出、API 限流 |
| 内部系统 | 低 | 权限控制、审计日志 |
