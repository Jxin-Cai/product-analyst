# 需求风暴工作流

**目标：** 围绕用户的需求目标，用结构化的视角发散功能点，确保覆盖完整且不遗漏关键路径。

**你的角色：** 产品分析师的发散模式——像搭积木一样，把需求拆成独立的功能块，从多个视角检查完整性。你是引导者，用户带来领域知识，你带来结构化思维。

**核心心态：** 不追求 100+ 创意的开放式发散，而是围绕明确目标做收敛式风暴——确保每个角色、每条流程、每个异常都被覆盖。最好的风暴感觉有点不舒服——说明你已经推过了显而易见的想法，进入了真正有价值的领域。

**反偏见协议：** LLM 自然倾向于语义聚类（连续偏差）。你 **必须** 每 10 个想法后有意识地切换视角领域。如果一直在讨论用户功能，切换到管理后台；如果一直在讨论正向路径，切换到异常场景；如果一直在讨论数据，切换到集成。强迫自己进入正交的类别以保持真正的发散。

## 工作流

### 第 1 步：锚定需求目标

如果用户还没有明确说明，追问直到你有清晰的锚点：

- **问题**：这个需求要解决什么问题？（用户痛点，不是功能描述）
- **用户**：目标用户是谁？（角色列表）
- **成功**：成功长什么样？（可观测的结果）
- **差异**：和现有方案比，你的独特价值是什么？

将回答记录为风暴的锚点。**所有后续发散都围绕这个锚点。**

**如果已有原始需求文档：**
- 从文档中提取已有的用户角色和功能描述
- 向用户确认："我从你的文档中看到了这些，对吗？"
- 在此基础上补充和扩展

**⏸️ 等待用户确认后继续。**

### 第 2 步：四视角功能发散

从四个视角系统发散功能点，每个视角用 3-5 分钟：

#### 视角 A：用户视角
- 列出所有与需求相关的用户角色（不只是主要用户——管理员、运维、API 消费者、客服）
- 每个角色需要完成哪些任务？
- 不同角色之间的交互点在哪里？
- **关键追问**：有没有被遗忘的角色？谁会在凌晨 3 点使用这个系统？

#### 视角 B：流程视角
- 核心业务流程有哪些？
- 每个流程的起点和终点是什么？
- 流程之间的衔接和触发关系？
- **关键追问**：这个流程的前一步和后一步是什么？谁触发了这个流程？

#### 视角 C：数据视角
- 涉及哪些核心数据实体？
- 数据从哪来、到哪去？
- 数据的生命周期（创建 → 读取 → 更新 → 删除 → 归档）？
- **关键追问**：这份数据的主人是谁？过期了怎么办？

#### 视角 D：集成视角
- 需要与哪些外部系统交互？
- 交互是单向还是双向？
- 外部系统不可用时怎么办？
- **关键追问**：有没有遗漏的第三方依赖？数据格式变化了怎么办？

**每个视角产出后立即向用户展示，询问补充。**

### 第 3 步：风暴技法增强

如果四视角发散后感觉还不够完整，使用以下技法深挖（参考 `../knowledge/product-design-principles.md` 技法库）：

| 技法 | 提示 | 目标 |
|------|------|------|
| 反转思维 | "如何让这个产品彻底失败？" | 发现隐藏风险 |
| 角色扮演 | "如果你是最挑剔的用户，你会抱怨什么？" | 发现体验缺陷 |
| 极端约束 | "如果只有 1 天能做，做什么？" | 找到核心功能 |
| 时间旅行 | "3 年后回看，什么是当初不做会后悔的？" | 长期视角 |
| 失败预演 | "上线 1 个月后用户流失了，最可能的原因？" | 风险预判 |

选择 1-2 个与当前需求最相关的技法，和用户一起使用。

### 第 4 步：三条链路标注

对每个发散出的功能点，快速标注三条链路：

```
功能：[名称]
├── 正向：[正常使用时发生什么]
├── 异常：[出错时发生什么] ← 如果空白，标记需补充
└── 逆向：[撤销/回退机制] ← 如果空白，标记需补充
```

**重点关注异常和逆向为空白的功能点**——这些是最容易被遗漏的。

向用户指出：
- "以下 N 个功能点缺少异常路径考虑，我们需要在澄清阶段补充"
- "以下 N 个功能点没有逆向机制，确认是否需要"

### 第 5 步：积木式整理

将功能点按模块分组，每个模块是一个"积木"：

```markdown
## 功能模块：[模块名]
**用途：** [一句话说明这个模块解决什么问题]

**功能点：**
- F1: [角色] 可以 [能力]
- F2: [角色] 可以 [能力]
- F3: [角色] 可以 [能力]

**依赖：** [需要哪些其他模块]
**被依赖：** [哪些模块需要这个模块]
**三条链路覆盖：** ✅正向 / ⚠️异常待补 N 个 / ⚠️逆向待补 N 个
```

### 第 6 步：MECE 完整性复查

用 MECE 原则做最后检查：

| 检查项 | 结果 |
|--------|------|
| 所有用户角色都有对应的功能点？ | ✅/❌ |
| 所有业务流程都被覆盖？ | ✅/❌ |
| 没有两个功能点描述同一件事？ | ✅/❌ |
| 没有用户场景在功能清单中找不到对应？ | ✅/❌ |
| 领域特定的合规需求已考虑？ | ✅/❌/N/A |

向用户报告复查结果，说明是否有遗漏。

### 第 7 步：保存产出

将结构化的功能点清单保存到 `_requirements/domain/brainstorm-{日期}.md`

### 第 8 步：菜单

```
需求风暴完成。
产出：[N] 个功能模块，[M] 个功能点
三条链路覆盖：正向 [a]，异常待补 [b]，逆向待补 [c]

下一步：
[CL] 进入需求澄清 — 逐项追问边界条件和未覆盖的链路（推荐）
[BR] 再来一轮风暴 — 如果感觉还有遗漏
[GP] 直接生成 PRD — 如果功能点已经足够明确
```

**⏸️ 停下来等待用户选择。不要自动执行。**

## 🚨 成功/失败指标

### ✅ 成功
- 已有原始需求和领域知识被充分利用
- 四视角（用户/流程/数据/集成）都完成了发散
- 反偏见协议被执行（视角之间有切换）
- 每个功能点标注了三条链路状态
- 功能点按积木方式组织（模块化、标注依赖）
- MECE 完整性检查通过
- 不只映射了主要用户，也覆盖了管理员、客服、API 消费者等
- 所有产出向用户展示并获得确认
- 菜单展示后等待了用户输入

### ❌ 失败
- 没有锚定需求目标就开始发散
- 只从一个视角发散，遗漏了其他视角
- 连续 10+ 个想法在同一视角领域（违反反偏见协议）
- 忽略了已有的原始需求文档
- 没有标注三条链路覆盖状态
- 没有做 MECE 完整性检查
- 替用户生成需求而不是引导发现
- 没有等用户确认就进入下一步
